# Debugging PX4 SITL from Neovim

Remote debugging via GDB — PX4 runs inside Docker, Neovim connects from the host.

---

## Prerequisites

### Host

- Neovim with [LazyVim](https://lazyvim.org) and `nvim-dap` installed
- GDB: `sudo apt-get install gdb`

### Neovim plugin

Copy `Tools/debug/dap-px4.lua` to your Neovim plugin directory:

```bash
cp Tools/debug/dap-px4.lua ~/.config/nvim/lua/plugins/dap-px4.lua
```

Restart Neovim. Mason will install the required tools automatically:

| Tool | Purpose |
|---|---|
| `clangd` | C++ LSP (completions, go-to-definition, diagnostics) |
| `cpptools` | DAP adapter driving GDB for debugging |

Or install them manually:
```
:MasonInstall clangd cpptools
```

### Docker image

`gdbserver` must be present in the image. It is already added to the `Dockerfile`.
If the image was built before this change, rebuild it:

```bash
docker build -t px4-sim-gz .
```

## Build

### From Neovim (recommended)

Press `<leader>B` from any buffer. This runs `Tools/build/make_sitl.sh`
asynchronously — Neovim stays responsive while the build runs in Docker.

- On success: a notification appears.
- On failure: errors and warnings are parsed and loaded into the **quickfix** list,
  which opens automatically. Navigate with `:cn` / `:cp`, or open trouble with
  `<leader>xq`.

### From the terminal

```bash
./Tools/build/make_sitl.sh             # px4_sitl_default (default)
./Tools/build/make_sitl.sh px4_sitl_default
```

The script runs the build inside Docker and translates container paths
(`/src/PX4-Autopilot/…`) back to host paths so Neovim quickfix can open files.

### First-time / manual build

If the Docker image isn't built yet, do an interactive build once:

```bash
./run_docker.sh          # shell inside the container
make px4_sitl_default    # build
exit                     # artifacts are on host via volume mount
```

## Running

### Option A — Docker (recommended)

**Terminal 1** — start the simulation stack:

```bash
./run_docker_debug.sh [model] [port]

# Examples:
./run_docker_debug.sh               # gz_x500, port 1234
./run_docker_debug.sh gz_x500_lidar
./run_docker_debug.sh gz_x500 5678
```

This builds the image if needed, starts Gazebo inside Docker, then launches
`gdbserver` listening on `localhost:<port>`. PX4 is paused until GDB connects.

**Neovim** — attach and start:

1. Open any C/C++ source file in the repo.
2. Press `<leader>dc` to open the configuration picker.
3. Select **PX4 SITL (Docker gdbserver :1234)**.
4. GDB connects and PX4 starts — Gazebo is already running in Docker.

### Option B — Local (no Docker)

Start Gazebo manually first:

```bash
gz sim -v4 -r default.sdf &
```

Then in Neovim select one of:

| Configuration | Description |
|---|---|
| `PX4 SITL (gz_x500)` | Fixed x500 model |
| `PX4 SITL (gz - pick model)` | Prompts for model name |
| `PX4 SITL (sihsim SYS_AUTOSTART=10040)` | Self-contained, no external sim |

## LSP (clangd)

The project root contains a `.clangd` file that:
- Points clangd to `build/px4_sitl_default/` for `compile_commands.json`
- Strips a few GCC-only flags that clangd rejects

`compile_commands.json` is generated by the build. Because the build runs in Docker,
paths inside it initially point to `/src/PX4-Autopilot/…`. The build script
(`Tools/build/make_sitl.sh`) rewrites them to host paths automatically after every
build, so clangd always gets correct paths.

If you ever see clangd complaining about missing files, re-run the build:
```
<leader>B
```

## GDB notes

The following GDB settings are applied automatically by the DAP config:

- **Pretty printing** enabled
- **SIGCONT** set to `nostop noprint nopass` — PX4's work-queue sends this signal
  internally; without this GDB would pause on every SIGCONT

## Troubleshooting

**`gdbserver: command not found` inside Docker**
Rebuild the image — `gdbserver` was added to the `Dockerfile` after the image was
last built: `docker build -t px4-sim-gz .`

**DAP picker shows no PX4 configurations**
`cpptools` is not installed. Run `:MasonInstall cpptools` and restart Neovim.

**GDB fails to connect**
Check that the container started and gdbserver is listening:
```bash
ss -tlnp | grep 1234
```
The container must be running before you trigger the DAP session.

**Gazebo not visible**
The container uses X11 forwarding. Make sure `xhost +` ran (the script does this)
and `$DISPLAY` is set correctly on the host.
